then analyzes the data to recommend: optimal gym times, optimal training days,
optimal foods, and when to rest.

**PHASE 3 ARCHITECTURE DECISIONS (Manager, 2026-07-08):**
- **EXTEND, not rewrite.** The vanilla-JS app is 9.5k lines of QA-hardened code
  (6 adversarial QA rounds) with deep conventions (OF.* module pattern, sw.js SHELL
  discipline, metric-internal storage) and shipped signed artifacts. A rewrite
  risks regressing the crown-jewel coach engine for zero user benefit. Social =
  new js modules in the same pattern; @supabase/supabase-js vendored as a UMD
  script file (no CDN at runtime, no build step, bundles into the native apps).
- **PRIVACY: private-by-default, social strictly OPT-IN.** No account required —
  every existing feature keeps working 100% on-device. Creating an account is an
  explicit choice; sharing is explicit per-post; coach/tracking data NEVER syncs;
  community benchmarks use only anonymized, k-anonymous aggregates from users who
  share posts/stats. The "your data never leaves your device" store claim is NO
  LONGER blanket-true once a user opts into social → privacy policy, Play Data
  safety, and Apple App Privacy forms must be rewritten honestly (Compliance task)
  to "local by default; optional social features send only what you choose".
- **Signature hook (DECIDED 2026-07-08, per Research-1): "RECEIPTS" — coach-verified
  achievement cards.** Every post is either a plain post or a Receipt: a visually
  distinct card whose numbers (e1RM sparkline, planned-vs-done consistency grid,
  weight-trend + learned-maintenance delta) are COMPUTED by the coach engine from
  longitudinal logged data — never typed by the user. Positioning: "Anyone can post
  a PR. OptimalFit proves it." Supporting mechanic #1 (MVP): **Weekly Receipts
  Day** — Sunday evening the engine auto-generates each user's Weekly Receipt and
  the network's receipts drop together (BeReal's sync power at a weekly cadence,
  avoiding its daily-fatigue decay; what you log Mon–Sat determines Sunday's card).
  Supporting #2 (v1.1, NOT MVP): Crew Leagues — small groups ranked on
  receipt-verified consistency only, with coach-excused misses (deload/illness).
  Rejected: daily BeReal-style moments (proven decay), global volume leaderboards
  (Strava's 4M-fake-activity lesson), Beli comparative ranking (no natural fit).
  Anti-cheat tiering: (1) history-depth gates (verified PR needs ≥6 logged sessions
  of the lift over ≥3 weeks), (2) server-side plausibility re-validation in a
  SECURITY DEFINER Postgres RPC that alone can set `verified` (e1RM weekly-jump
  caps, duration/set sanity, monotonic timestamps) — client can never write the
  flag, (3) small-crew social visibility. Manager scope trims vs research proposal:
  RPC validation instead of Edge Functions (no extra Deno runtime), NO device-key
  HMAC for MVP (key would live client-side anyway — theater), Receipts Day is
  in-app only (no push infra exists). Verified receipts double as the CLEAN corpus
  for anonymized community benchmarks (the coach↔social flywheel). Full research
  report (Beli/Strava/BeReal/Duolingo/Whoop/Hevy/GymRats mechanics + sources) in
  Research-1's Loop Log row.

- Builder-2 notes (Iteration 2 — insights engine + charts):

  - Engine entry: `OF.engine.analyzeAll({sleep,food,exercise,body})` → `{plan, readiness, timeOfDay, weekdays, sleep, rest, food, trends}`. Every insight is `{status:"ok"|"insufficient", message?...}`; insufficient always says how much more data is needed. No NaN/undefined can reach the UI (verified by regex over rendered text).
  - IMPORTANT design decision — sleep adjustment: raw group averages on the demo seed are confounded (by chance, 3rd-streak-day and 19-24h workouts followed unusually good sleep, which masked the fatigue pattern and made "evening" look best). `buildSleepAdjuster()` fits perf ~ sleepHours (needs ≥8 pairs) and group RANKINGS (time-of-day, weekdays, consecutive-day fatigue) use the sleep-adjusted residuals; UI shows raw averages + a "ranking adjusted for prior-night sleep" chip (rest card shows raw AND adj). With this, all demo patterns are recovered: morning best (3.8 vs 3.1 afternoon), 7h+ sleep 4/5 vs 3/5 (r=0.76), carb-rich pre-meal 3.7 vs 3.2, fatigue drop on 3rd+ day → max 2 consecutive → weekly plan "Train Tue/Thu/Fri mornings; rest Mon/Wed/Sat/Sun; carb-rich pre-meal; 7.5h sleep".
  - Fatigue detection threshold: sleep-adjusted drop ≥ 0.25 vs weighted day-1/day-2 baseline (raw 0.4 threshold never fired on the confounded demo data; 0.25 ≈ 1 SE at these sample sizes — heuristic, tagged with confidence).
  - Time buckets: morning 05-11, midday 11-15, afternoon 15-19, evening 19-24; 00-05 folds into evening. Min 3 sessions/bucket for a recommendation; confidence low/<5/medium/<10/high. Weekdays need ≥2 sessions on ≥2 days.
  - "Pre-workout meal" = any food record same date within 3h before startTime (per spec). On demo data almost every workout has one (lunch/snacks count), so the with/without comparison is weak (34 vs 4, "even", low confidence) — the carb comparison (≥40g vs <40g among pre-meals) is where the demo pattern shows. Not a bug, but QA should know.

- Builder-5 notes (Iteration 7 — PWA, phone mode, health import):
  - **Pairing-code design (kept deliberately simple):** one 6-digit code per server RUN (regenerated at every restart, `secrets.randbelow`), compared with `compare_digest`, sent as `X-OF-Key` on every /api request by coach.js when stored. It gates ONLY /api/coach (the expensive/private endpoint); static files and /api/health stay open to the LAN — health leaks nothing but `{ok, claude, phoneMode, lanUrls, keyOk}`. `keyOk` in /api/health exists so the app can validate a code without burning an LLM call. A restarted server = new code; the app detects the 401, wipes `optimalfit.pairKey` and re-shows the pairing card. NOT defense-grade: no rate limit on code guesses (10^6 space, LAN-only, gates a chat endpoint — accepted), no TLS (LAN http; hence also NO service worker on the phone — SW needs a secure context, so the phone PWA is an installed shortcut without offline cache; localhost/https gets the full offline shell).
  - **Apple parser approach:** File.slice → arrayBuffer in 4 MB chunks, `TextDecoder(stream:true)` (multibyte chars split across chunks are safe), keep the partial last line as carry, regex per complete line (`indexOf('<Record')` prefilter, then type-substring checks before attribute regexes). `setTimeout(0)` between chunks keeps the tab responsive; progress bar per chunk. Measured: 20 MB / 96k records ≈ **0.48 s** in the preview browser. Timezone suffixes in start/endDate are intentionally ignored (record-local wall time is what "that day" means to the user). BodyMassIndex explicitly excluded from the BodyMass match. Sleep: interval end-date = night's record date; bed = earliest interval start (can be a previous-day nap in pathological data — accepted), duration = sum of Asleep* intervals only (InBed/Awake dropped), nights <10 min or >20 h dropped (matches the form's 20 h cap).
  - **Import merge details:** steps update-if-day-exists (regardless of origin — the health app is considered authoritative for a day's total); water updates only when the day has exactly ONE existing record, skips days with 2+ (those are the user's manual quick-adds; can't collapse them without destroying data); body/sleep skip logged days entirely. All writes via OF.storage.add/update → normal id/createdAt flow; a saveAll failure aborts the loop and says counts are incomplete.
  - **Fixed in passing:** serve.py stdout now `line_buffering=True` (banner incl. pairing code was invisible when stdout was redirected to a file, e.g. by a service wrapper or test harness; console use unaffected).
  - **Known gaps / limits:** NO live HealthKit/Samsung sync — file import only (Apple exposes no web API; this is a browser app). Phone PWA over LAN http = no service worker → no offline, and each device's localStorage is separate (Export/Import to move data; stated honestly in Settings). Pairing code entry appears only on the Coach tab (other tabs never need it). Water import can't merge into days with multiple manual entries (skipped + reported). Samsung CSV day_time is interpreted as UTC midnight (matches observed exports; a wrong-TZ edge would shift a day at most). export.zip is detected and explained but NOT unzipped in-browser (stdlib-only constraint — no zip lib worth hand-rolling for this). iOS "Add to Home Screen" uses apple-touch-icon + metas; iOS ignores most of the manifest but standalone display works. Icons: regenerate via tools/make_icons.py if the accent color ever changes.
